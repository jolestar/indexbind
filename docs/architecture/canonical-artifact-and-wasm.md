# Canonical Artifact And WASM Architecture

This document describes the intended next-stage architecture for `indexbind`.

The project boundary is intentionally narrow:

- build a retrieval artifact offline
- query that artifact across Node and web runtimes
- return ranked document matches

`indexbind` should not own document loading at runtime. Applications should use search results to locate or render content through their own storage and routing layers.

## Architecture Direction

The architecture moves from a native-first SQLite implementation to a host-neutral retrieval model:

1. Refactor the Rust core so retrieval logic is separate from storage and host integration.
2. Define a canonical file artifact as the product-level artifact contract.
3. Implement a WASM query runtime on top of that canonical artifact.
4. Expose consistent TypeScript APIs for build and query.
5. Treat SQLite as an optional native optimization layer, not the core product format.

## Responsibilities

`indexbind` is responsible for:

- accepting normalized documents as build input
- producing a reusable retrieval artifact
- running lexical, vector, and reranked search
- returning stable document identifiers and match metadata

`indexbind` is not responsible for:

- reading source files at runtime
- serving document bodies
- defining application routing
- managing site deployment paths

## Document Model

The long-term build input contract should use a host-neutral document shape:

```ts
type JsonValue =
  | null
  | boolean
  | number
  | string
  | JsonValue[]
  | { [key: string]: JsonValue };

interface BuildDocument {
  docId?: string;
  relativePath: string;
  canonicalUrl?: string;
  title?: string;
  summary?: string;
  content: string;
  metadata?: Record<string, JsonValue>;
}
```

Field semantics:

- `relativePath` is the stable logical path inside the artifact.
- `canonicalUrl` is the application-facing URL used for navigation.
- `docId` is either supplied by the caller or deterministically generated by the build layer.
- `metadata` can contain structured JSON, but initial filtering only needs to support top-level scalar equality.

The old `originalPath` field should not remain part of the public runtime contract. If the build process needs local source provenance, that can exist internally as optional debug-only `sourcePath` metadata.

## Search Result Contract

Runtime search results should be independent from the local filesystem:

```ts
interface SearchHit {
  docId: string;
  relativePath: string;
  canonicalUrl?: string;
  title?: string;
  summary?: string;
  metadata: Record<string, JsonValue>;
  score: number;
  bestMatch: {
    chunkId: string | number;
    excerpt: string;
    headingPath: string[];
    charStart: number;
    charEnd: number;
    score: number;
  };
}
```

There is no required `readDocument()` capability in the core contract. Applications can use `docId`, `relativePath`, `canonicalUrl`, or custom metadata to retrieve full content on their own.

## Canonical File Artifact

The canonical artifact should be a directory bundle. It is the single authority for cross-runtime compatibility.

Required files:

- `manifest.json`
- `documents.json`
- `chunks.json`
- `vectors.bin`
- `postings.json`

Optional files:

- `model/`
- `extra/`

### manifest.json

Recommended fields:

- `schemaVersion`
- `artifactFormat`
- `builtAt`
- `embeddingBackend`
- `documentCount`
- `chunkCount`
- `vectorDimensions`
- `chunking`
- `files`
- `features`

### documents.json

Each document record should contain:

- `docId`
- `relativePath`
- `canonicalUrl?`
- `title?`
- `summary?`
- `metadata`
- `firstChunkIndex`
- `chunkCount`

### chunks.json

Each chunk record should contain:

- `chunkId`
- `docId`
- `ordinal`
- `headingPath`
- `charStart`
- `charEnd`
- `tokenCount`
- `excerpt`
- `chunkText`

`chunkText` should stay in the initial format to simplify lexical retrieval and reranking. Space optimization can be handled later if needed.

### vectors.bin

- stores chunk vectors in `Float32Array` order
- ordering is defined by `chunks.json`
- vector dimensions are defined by `manifest.json`

### postings.json

This file is the lexical index used outside SQLite. The minimum contract is:

- token to posting list mapping
- each posting contains `chunkIndex` and `termFrequency`
- global statistics needed for BM25

The initial tokenizer can stay simple and aligned with the current implementation:

- lowercase
- split on non-alphanumeric characters
- no stemming
- no stopword removal

This keeps native and WASM behavior easier to compare during the transition.

## Runtime Strategy

### Node

- Node continues to provide the primary runtime experience.
- During the transition, Node may support both the current native SQLite path and the new canonical file path.
- Once the canonical artifact is stable, the query engine should use the same retrieval logic across native and WASM.

### Browser And Worker

- Browser and Worker runtimes should consume the canonical file artifact directly.
- They should support `open`, `info`, and `search`.
- They should not depend on local filesystem access.
- They should not depend on SQLite or FTS5.

## Rust Refactor Plan

The Rust implementation should be split into three layers:

### 1. Canonical model layer

Owns:

- artifact metadata types
- document and chunk types
- query options
- search result types

### 2. Query engine layer

Owns:

- vector similarity scoring
- lexical scoring
- reciprocal rank fusion
- reranker logic
- result shaping

This layer must not depend on:

- `rusqlite`
- `std::fs`
- NAPI bindings

### 3. Adapter layer

Owns:

- current SQLite artifact access
- current FTS5 lexical search path
- local model loading
- current CLI directory scanning

This keeps the long-term query engine portable without forcing every existing native implementation to disappear immediately.

## model2vec And WASM

The hardest part of web parity is not the retrieval logic. It is `model2vec`.

Current native retrieval uses `model2vec-rs` for:

- query embedding
- chunk embedding at build time
- embedding-based reranking

There is no mature, ready-made pure JavaScript runtime path to reuse directly. The realistic direction is:

- keep build-time embedding in native Rust
- validate whether query-time `model2vec` can run in WASM with acceptable size and startup cost
- only promise web parity after that spike succeeds

Because of that, the execution order should be:

1. run a focused `model2vec` in WASM spike
2. refactor the Rust core around a portable query engine
3. implement the WASM query runtime
4. expose the TypeScript web API

If the spike fails, the architecture should adapt before further implementation work continues.

## SQLite Position

SQLite and FTS5 served the current native implementation well because they solved:

- single-file artifact persistence
- lexical retrieval
- fast progress for the initial prototype

They should not remain the public artifact contract for the long term.

Future role:

- optional native compiled artifact
- optional storage/performance optimization
- never the only cross-runtime format

That means semantic behavior should be defined by the canonical file artifact and shared query engine, not by a SQLite schema.

## API Surface Direction

The TypeScript package should eventually expose three clear entry points:

- `indexbind/build`
- `indexbind/node`
- `indexbind/web`

The public contract should center on:

- build from normalized documents
- open an artifact
- inspect artifact metadata
- search

It should not center on runtime file loading.

## Acceptance Criteria For The New Architecture

The new direction is complete when:

- build input is programmatic and host-neutral
- canonical artifacts can be generated deterministically
- Node and WASM runtimes return equivalent ranking behavior on shared fixtures
- search results do not rely on local absolute paths
- applications can integrate `indexbind` as a retrieval layer without coupling to the builder's filesystem
